# Skills market and connector architecture

## Decision

Evotown should not become the package manager for OpenClaw, Hermes, SkillLite, or custom runtimes.

The enterprise architecture has three distinct layers:

1. **Private Skills Market** — source of reusable skill assets and bootstrap bundles.
2. **Connector / proxy node** — integration and governance layer near runtimes.
3. **Evotown** — observability, evaluation, policy, review, and culture layer.

Runtimes install skills from the market, execute locally, and report normalized state through connectors into Evotown.

## Product roles

### Private Skills Market

The market owns skill distribution and lifecycle.

Responsibilities:

- host company-approved bootstrap skill bundles
- store team-shared skills and reviewed evolved skills
- manage versions, signatures, metadata, tags, and permissions
- expose install manifests for OpenClaw, Hermes, SkillLite, and custom runtimes
- receive candidate skills from connectors or Evotown review workflows
- keep rollback pointers and deprecation state

The market is the system of record for reusable skill packages. Evotown may display market metadata, but it should not be the installer or package registry itself.

### Connector / proxy node

The connector is the runtime integration layer. It may run centrally, per team, per workspace, or near employee machines.

Responsibilities:

- normalize OpenClaw, Hermes, SkillLite, and custom runtime events
- attach `tenant_id`, `team_id`, `agent_id`, `engine_id`, and `task_id`
- send `run.started`, `run.progress`, and `run.completed` events to Evotown
- redact secrets and filter artifacts before upload
- enforce or cache policy from Evotown when policy pull exists
- collect candidate skills generated by runtimes
- submit candidate skills to the private Skills Market for review
- optionally proxy market install manifests to runtimes in locked-down networks

The connector is not a new agent runtime and should not own long-term skill lifecycle state.

### Evotown

Evotown owns control-plane visibility and evidence.

Responsibilities:

- register engines and connectors
- store run summaries, run events, artifacts, signals, and policy violations
- visualize live agent activity and team-level outcomes
- link candidate assets to source runs and evaluation results
- support review / approval workflows
- show skill reuse impact after market promotion

Evotown is the system of record for run evidence, not for package installation.

## Bootstrap flow

```text
Enterprise admin defines bootstrap bundle
-> Private Skills Market signs and publishes bundle
-> OpenClaw / Hermes / SkillLite runtime installs bundle
-> Runtime executes tasks locally
-> Connector normalizes lifecycle events
-> Evotown receives run.started / run.progress / run.completed
```

Minimum runtime configuration:

```yaml
skills_market:
  url: https://skills.company.internal
  bootstrap_bundle: default-agent-skills
  channel: stable

evotown_connector:
  endpoint: https://agent-connector.company.internal
  tenant_id: company-a
  team_id: growth-team
  agent_id: hermes-agent-001
```

## Skill evolution sharing flow

```text
Runtime generates or improves a local skill
-> Connector detects candidate skill
-> Connector sends candidate to Skills Market
-> Automated validation runs
-> Reviewer approves / rejects
-> Approved skill becomes team or company visible
-> Runtimes install by manifest or recommendation
-> Evotown measures reuse and outcome impact
```

Rules:

1. Locally generated skills must not become company-wide defaults automatically.
2. Every candidate skill must reference a source `run_id`, author, team, and runtime.
3. A candidate can be visible to reviewers before it is installable by default.
4. Promotion requires evidence: validation result, source run, review decision, and rollback pointer.
5. Evotown may recommend approved skills, but installation still goes through the market.

## Minimal Skills Market API

This is a product contract direction, not yet a required Evotown backend implementation.

### `GET /api/v1/skill-bundles/{bundle_id}/manifest`

Returns an install manifest for a runtime.

Minimum fields:

- `bundle_id`
- `version`
- `channel`
- `runtime_targets`: `openclaw`, `hermes`, `skilllite`, `custom`
- `skills`: array of skill package references
- `signature`
- `published_at`

### `GET /api/v1/skills`

Searches approved or visible skills.

Filters:

- `team_id`
- `runtime_target`
- `tag`
- `status`
- `query`

### `POST /api/v1/skill-candidates`

Creates a reviewable candidate skill.

Minimum fields:

- `candidate_id`
- `source_run_id`
- `tenant_id`
- `team_id`
- `agent_id`
- `engine_id`
- `runtime_target`
- `name`
- `description`
- `package_url` or `inline_manifest`
- `signals`

### `POST /api/v1/skill-candidates/{candidate_id}/review`

Approves or rejects a candidate.

Minimum fields:

- `decision`: `approved` or `rejected`
- `reviewer`
- `reason`
- `visibility`: `private`, `team`, or `company`
- `promotion_channel`: optional, e.g. `stable`, `beta`

## Connector to Evotown event shape

Connectors should use the lifecycle ingest shape:

```json
{
  "event_type": "run.started",
  "run_id": "xxx",
  "tenant_id": "company-a",
  "team_id": "growth-team",
  "agent_id": "agent-001",
  "engine_id": "hermes-local",
  "engine_type": "hermes",
  "engine_version": "1.4.0",
  "task_id": "task-xxx",
  "status": "running",
  "ts": "2026-05-25T09:00:00Z",
  "signals": {}
}
```

Runtime-specific details belong in `payload`; cross-runtime fields should stay top-level.

## Non-goals

- Evotown does not install packages into employee machines directly.
- The market does not store raw run telemetry as its primary record.
- Connectors do not bypass review by publishing local skills directly to company scope.
- SkillLite-specific file layout must not be required for OpenClaw or Hermes integrations.
